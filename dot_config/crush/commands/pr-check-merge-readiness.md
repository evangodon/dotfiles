# PR Merge Readiness

Check which of your open PRs are ready to merge.

## Step 1: Fetch open PRs

Fetch all open, non-draft, unmerged PRs authored by the current user:

```bash
gh search prs --author=@me --state=open --draft=false --limit=50 --json number,title,repository,url,createdAt 2>&1
```

Note: `gh search prs --json` does not support `headRefName`/`baseRefName`. Fetch branch names separately per PR using:

```bash
gh pr view {number} --repo {owner/repo} --json headRefName,baseRefName --jq '{head: .headRefName, base: .baseRefName}' 2>&1
```

Run these in parallel for all PRs.

## Step 2: Display table and ask which to check

Present the results in a numbered table:

| # | Repo | PR | Branch | Created |
|---|------|----|--------|---------|

Ask: "Which PRs would you like to check? (e.g. '1, 3' or 'all')"

Wait for the user's response before continuing.

## Step 3: For each selected PR, check merge readiness

### A) CI status

```bash
gh pr checks {number} --repo {owner/repo} 2>&1
```

- All pass → ✅ CI passing
- Any fail → ❌ CI failing (list the failing checks by name)
- Any pending → ⏳ CI still running

### B) Branch freshness

Get the default branch:
```bash
gh repo view {owner/repo} --json defaultBranchRef --jq '.defaultBranchRef.name' 2>&1
```

Check how far behind:
```bash
gh api repos/{owner}/{repo}/compare/{defaultBranch}...{headBranch} --jq '{behind: .behind_by, ahead: .ahead_by}' 2>&1
```

- `behind_by` == 0 → ✅ Up to date
- `behind_by` > 0 → ⚠️ Behind by N commits

### C) Review decision & unresolved comments

```bash
gh api graphql -f query='
  query($owner: String!, $repo: String!, $number: Int!) {
    repository(owner: $owner, name: $repo) {
      pullRequest(number: $number) {
        reviewDecision
        reviewThreads(first: 100) {
          nodes {
            isResolved
          }
        }
      }
    }
  }
' -f owner={owner} -f repo={repo} -F number={number} --jq '{
  reviewDecision: .data.repository.pullRequest.reviewDecision,
  unresolved: [.data.repository.pullRequest.reviewThreads.nodes[] | select(.isResolved == false)] | length
}'
```

- `reviewDecision` is `APPROVED` → ✅ Approved
- `reviewDecision` is `CHANGES_REQUESTED` → ❌ Changes requested — also fetch the review comments to check if they are all outdated (addressed). If all comments are outdated, note this and recommend re-requesting review.
- `reviewDecision` is `REVIEW_REQUIRED` or null → ⏳ Awaiting review
- `unresolved` > 0 → ⚠️ N unresolved comment threads (report count only, do not attempt to fix)

To check if CHANGES_REQUESTED comments are outdated:
```bash
gh api graphql -f query='
  query($owner: String!, $repo: String!, $number: Int!) {
    repository(owner: $owner, name: $repo) {
      pullRequest(number: $number) {
        reviews(last: 20) {
          nodes {
            author { login }
            state
            body
            submittedAt
            comments(first: 20) {
              nodes {
                body
                path
                outdated
              }
            }
          }
        }
      }
    }
  }
' -f owner={owner} -f repo={repo} -F number={number} --jq '.data.repository.pullRequest.reviews.nodes[] | select(.state == "CHANGES_REQUESTED") | {author: .author.login, comments: [.comments.nodes[] | {outdated: .outdated, body: .body[0:80]}]}' 2>&1
```

### D) PR description accuracy

```bash
gh pr view {number} --repo {owner/repo} --json title,body,commits 2>&1
gh pr diff {number} --repo {owner/repo} --name-only 2>&1
```

Read the description and cross-check against the changed files and commit messages. Flag if:
- There are unresolved TODOs in the description
- The description references things not reflected in the diff
- The description is empty or a template placeholder

## Step 4: Show summary for all checked PRs

For each PR, display:

```
## PR #123 — Title (owner/repo)

| Check | Status | Details |
|-------|--------|---------|
| CI | ✅ Passing | All 5 checks passed |
| Branch | ⚠️ Behind | Behind master by 3 commits |
| Review | ✅ Approved | |
| Unresolved comments | ⚠️ 2 | 2 unresolved threads |
| Description | ✅ Accurate | Matches changes |
```

Check ALL selected PRs before offering any fixes.

Then offer next actions:
- **Behind base**: offer to rebase and push. When rebasing:
  - **Never clone the repo.** If the local directory is not known, ask the user where it is before proceeding.
  - Run `git fetch origin` first and use `origin/master` (or the default branch) as the rebase target — do not assume the local master is up to date.
  - Resolve conflicts in generated files (e.g. `generated/generated.go`) by taking master's version (`git checkout --theirs`) since those files are regenerated by the build.
  - Use `GIT_EDITOR=true git rebase --continue` to avoid interactive editor prompts.
  - Force-push with `git push --force-with-lease` after a successful rebase.
- **CI failing**: note that the failure may be flaky/infrastructure — check if it looks related to the code changes before offering to investigate; if it is, use the `debug-ci-failure` command
- **Description inaccurate**: offer to update it
- **Changes requested (all comments outdated)**: note that the review is stale — all comments were addressed in subsequent commits. Recommend re-requesting review from the reviewer.
